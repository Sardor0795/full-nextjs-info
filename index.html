What is Next.js?

Next.js is a React framework built by Vercel that gives you everything you need for production: server-side rendering, static generation, API routes, file-based routing, and more — out of the box. Instead of configuring webpack, babel, routing libraries, and SSR yourself, Next.js handles it all.

Why use Next.js over plain React?

// Plain React (Create React App / Vite):
// - Client-side only rendering (bad for SEO)
// - No built-in routing (need react-router)
// - No server-side code (need separate backend)
// - Manual code splitting
// - No image/font optimization

// Next.js gives you ALL of this built-in:
// ✓ Server-Side Rendering (SSR) & Static Generation (SSG)
// ✓ File-based routing (no react-router needed)
// ✓ API routes / Route handlers (full-stack)
// ✓ Automatic code splitting per route
// ✓ Image, font, script optimization
// ✓ Server Components (React 19)
// ✓ Server Actions (form mutations)
// ✓ Middleware, caching, ISR
Version note: Next.js has two routing systems. The Pages Router (legacy, pre-13) and the App Router (Next.js 13+, recommended). This course uses the App Router exclusively.

Key Takeaways

  • Next.js = React + routing + SSR + API + optimizations in one framework
  • Built and maintained by Vercel — the leading React deployment platform
  • Use the App Router for all new projects (Pages Router is legacy)
  • Full-stack: write frontend AND backend code in one project

Next.js uses file-system conventions. Specific file names have special meaning inside the app/ directory: 

my-app/
├── src/
│   └── app/
│       ├── layout.tsx      ← Root layout (wraps all pages)
│       ├── page.tsx        ← Home page (/)
│       ├── loading.tsx     ← Loading UI (Suspense)
│       ├── error.tsx       ← Error boundary
│       ├── not-found.tsx   ← 404 page
│       ├── globals.css     ← Global styles
│       ├── about/
│       │   └── page.tsx    ← /about
│       ├── blog/
│       │   ├── page.tsx    ← /blog
│       │   └── [slug]/
│       │       └── page.tsx ← /blog/my-post
│       └── api/
│           └── users/
│               └── route.ts ← API: /api/users
├── public/                 ← Static files (images, etc.)
├── next.config.ts          ← Next.js configuration
├── tailwind.config.ts      ← Tailwind config
├── tsconfig.json           ← TypeScript config
└── package.json

Special file names in the App Router:

// page.tsx    → Makes a route publicly accessible
// layout.tsx  → Shared UI wrapper (persists across navigation)
// loading.tsx → Automatic loading state (uses React Suspense)
// error.tsx   → Error boundary for the route segment
// not-found.tsx → 404 page for the segment
// route.ts    → API endpoint (GET, POST, etc.)
// template.tsx → Like layout but re-mounts on navigation
// default.tsx → Fallback for parallel routes

Key Takeaways

  • Folders define routes, page.tsx makes them accessible
  • layout.tsx wraps pages and persists state across navigation
  • Special files (loading, error, not-found) provide built-in UX patterns
  • The public/ folder serves static assets at the root URL

Pages & Basic Routing

In Next.js, folders create URL segments and page.tsx makes them navigable. No router configuration needed. 

// app/page.tsx → accessible at /
export default function HomePage() {
  return <h1>Welcome to my app</h1>;
}

// app/about/page.tsx → accessible at /about
export default function AboutPage() {
  return <h1>About us</h1>;
}

// app/blog/posts/page.tsx → accessible at /blog/posts
export default function PostsPage() {
  return <h1>Blog posts</h1>;
}

Root layout is required — it wraps every page: 

// app/layout.tsx — required root layout
export default function RootLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en">
      <body>
        <nav>My App</nav>
        {children}
      </body>
    </html>
  );
}

Key Takeaways

  • Folder = URL segment, page.tsx = renders the route
  • Nested folders create nested URLs automatically
  • layout.tsx is required at the root; nested layouts are optional
  • Pages are Server Components by default — they can be async

Linking & Navigation

Next.js provides the <Link> component for client-side navigation. It prefetches routes automatically for instant transitions. 

import Link from "next/link";

export default function Nav() {
  return (
    <nav>
      <Link href="/">Home</Link>
      <Link href="/about">About</Link>
      <Link href="/blog/my-post">Post</Link>

      {/* Prefetch disabled for rarely visited pages */}
      <Link href="/settings" prefetch={false}>Settings</Link>
    </nav>
  );
}

Programmatic navigation with useRouter (client components only): 

"use client";
import { useRouter, usePathname, useSearchParams } from "next/navigation";

export function SearchButton() {
  const router = useRouter();
  const pathname = usePathname();     // current path
  const params = useSearchParams();   // query params

  return (
    <button onClick={() => {
      router.push("/search?q=nextjs");   // navigate
      router.replace("/login");          // no history entry
      router.back();                      // go back
      router.refresh();                   // re-fetch server data
    }}>Navigate</button>
  );
}
Next.js 13+ change: Navigation hooks moved from "next/router" (Pages Router) to "next/navigation" (App Router). The API is different — useRouter() no longer has query or asPath.

Key Takeaways

  • <Link href="..."> for declarative, prefetched navigation
  • useRouter() from "next/navigation" for programmatic navigation
  • usePathname() returns current path, useSearchParams() returns query
  • Links prefetch visible routes by default for instant navigation

Metadata & SEO

Next.js has a built-in Metadata API for managing <head> tags — titles, descriptions, Open Graph images, and more. 

// app/layout.tsx — static metadata
import type { Metadata } from "next";

export const metadata: Metadata = {

  openGraph: {
    title: "My App",
    description: "The best app ever built",
    url: "https://myapp.com",
    images: ["/og-image.png"],
  },
};

Dynamic metadata for pages with data:

// app/blog/[slug]/page.tsx
import type { Metadata } from "next";

export async function generateMetadata(
  { params }: { params: Promise<{ slug: string }> }
): Promise<Metadata> {
  const { slug } = await params;
  const post = await getPost(slug);
  return {
    title: post.title,
    description: post.excerpt,
    openGraph: { images: [post.coverImage] },
  };
}
Next.js 15 change: params in generateMetadata is now a Promise — you must await it. In Next.js 14, it was a plain object.

Key Takeaways

  • Export metadata object for static metadata or generateMetadata() for dynamic
  • Use title.template in layout to auto-format child page titles
  • Open Graph, Twitter cards, icons, and robots are all supported
  • Metadata is automatically deduped and merged from layouts → pages

Dynamic Routes

Wrap folder names in brackets to create dynamic URL segments. The parameter value is passed as a prop to the page. 

// app/users/[id]/page.tsx → /users/1, /users/42
export default async function UserPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  const user = await getUser(Number(id));
  return <h1>{user.name}</h1>;
}

// app/blog/[...slug]/page.tsx → catches ALL sub-segments
// /blog/2024/jan/my-post → slug = ["2024", "jan", "my-post"]

// app/docs/[[...slug]]/page.tsx → OPTIONAL catch-all
// /docs → slug = undefined
// /docs/getting-started → slug = ["getting-started"]

Generate static params for static generation: 

export async function generateStaticParams() {
  const posts = await getAllPosts();
  return posts.map((post) => ({ slug: post.slug }));
}
// Pre-renders /blog/first-post, /blog/second-post, etc.
Next.js 15: params is now async (a Promise). You must await params before accessing values. In Next.js 14, it was synchronous.

Key Takeaways

  • [id] = single dynamic segment, [...slug] = catch-all, [[...slug]] = optional catch-all
  • Params are always strings — parse numbers with Number(id)
  • generateStaticParams() pre-renders dynamic pages at build time
  • In Next.js 15+, params is a Promise — always await it

Layouts & Templates

Layouts wrap pages and persist across navigation — the component doesn't unmount when you navigate between child pages. This preserves state (like scroll position or input values). 

// app/dashboard/layout.tsx — wraps all /dashboard/* pages
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex">
      <aside>
        <nav>Dashboard Sidebar</nav>
      </aside>
      <main>{children}</main>
    </div>
  );
}

Templates are like layouts but they re-mount on every navigation (create a new instance). Use them for enter/exit animations or per-page logging: 

// app/dashboard/template.tsx
export default function DashboardTemplate({
  children,
}: {
  children: React.ReactNode;
}) {
  // This runs on EVERY navigation within /dashboard
  return <div className="animate-fadeIn">{children}</div>;
}

Key Takeaways

  • Layout = persists across navigation (keeps state), Template = re-mounts each time
  • Layouts can be nested: root layout → dashboard layout → settings layout
  • Root layout is required and must include <html> and <body>
  • Layouts are Server Components by default — can fetch data directly

Wrap folder names in parentheses to organize routes without affecting the URL. The folder name is excluded from the path.

├── (marketing)/

│   ├── page.tsx      ← / (home page)
│   ├── about/
│   │   └── page.tsx  ← /about
│   └── pricing/
│       └── page.tsx  ← /pricing
├── (app)/
│   ├── layout.tsx    ← App layout (sidebar + topbar)

│   │   └── page.tsx  ← /dashboard

Parallel & Intercepting Routes

Parallel routes let you render multiple pages simultaneously in the same layout, using named slots (@slotName): 

// app/dashboard/layout.tsx
export default function Layout({
  children,
  analytics,
  team,
}: {
  children: React.ReactNode;
  analytics: React.ReactNode;
  team: React.ReactNode;
}) {
  return (
    <div>
      {children}
      {analytics}  {/* from @analytics/page.tsx */}
      {team}        {/* from @team/page.tsx */}
    </div>
  );
}

// Folder structure:
// app/dashboard/@analytics/page.tsx
// app/dashboard/@team/page.tsx

Intercepting routes let you load a route within the current layout (like modals). Use (.) to intercept a sibling, (..) for parent: 

// app/feed/@modal/(.)photo/[id]/page.tsx
// When clicking a photo link from /feed, it opens in a modal
// Direct navigation to /photo/123 shows the full page instead

// Convention:
// (.)  — intercept same level
// (..) — intercept one level up
// (..)(..)} — intercept two levels up
// (...) — intercept from root

Key Takeaways

  • Parallel routes (@slot) render multiple pages in one layout simultaneously
  • Intercepting routes show content in context (modals) without full navigation
  • The Instagram-style photo modal pattern is the classic use case
  • Use default.tsx as fallback when a parallel slot has no matching route

Loading & Error States

Next.js provides built-in loading and error handling through special files. loading.tsx creates an automatic Suspense boundary around the page. 

// app/dashboard/loading.tsx — shows while page loads
export default function Loading() {
  return (
    <div className="animate-pulse">
      <div className="h-8 bg-gray-200 rounded w-1/3 mb-4" />

      <div className="h-4 bg-gray-200 rounded w-2/3" />

Error boundaries catch errors and show a fallback UI: 

"use client"; // error.tsx must be a client component

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    <div>
      <h2>Something went wrong!</h2>
      <p>{error.message}</p>
      <button onClick={reset}>Try again</button>
    </div>
  );
}

Key Takeaways

  • loading.tsx = automatic Suspense boundary (shows skeleton while loading)
  • error.tsx = automatic error boundary (must be a Client Component)
  • not-found.tsx = custom 404 page, triggered by notFound() function
  • These are scoped per segment — each folder can have its own loading/error UI

Redirects & Rewrites

Next.js offers multiple ways to redirect users — in components, middleware, or config. 

// In Server Components or Server Actions
import { redirect, permanentRedirect, notFound } from "next/navigation";

export default async function Page() {
  const user = await getUser();

  if (!user) redirect("/login");            // 307 temporary
  if (user.banned) permanentRedirect("/banned"); // 308 permanent
  if (!user.active) notFound();              // shows not-found.tsx

  return <h1>Welcome {user.name}</h1>;
}

Config-based redirects and rewrites in next.config.ts: 

// next.config.ts
const nextConfig = {
  async redirects() {
    return [
      { source: "/old-blog/:slug", destination: "/blog/:slug", permanent: true },
    ];
  },
  async rewrites() {
    return [
      { source: "/api/proxy/:path*", destination: "https://api.example.com/:path*" },
    ];
  },
};

Key Takeaways

  • redirect() for temporary (307), permanentRedirect() for permanent (308)
  • notFound() triggers the nearest not-found.tsx
  • Config-based redirects run at the edge before the page loads
  • Rewrites map URLs without the user seeing the change (proxy pattern)

React Server Components (RSC)

In Next.js App Router, components are Server Components by default. They run on the server, can be async, and send zero JavaScript to the client. 

// This is a Server Component — no "use client" needed
// It can directly query a database, read files, call APIs
export default async function UsersPage() {
  // This runs on the server only — never exposed to the browser
  const users = await db.user.findMany();

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}

// What you CAN'T do in Server Components:
// ✗ useState, useEffect, useRef (no React hooks)
// ✗ onClick, onChange (no event handlers)
// ✗ browser APIs (window, document, localStorage)

Key Takeaways

  • Server Components are the default — no directive needed
  • They can be async and directly access databases, file system, secrets
  • They send zero JS to the client — only the rendered HTML
  • Cannot use hooks, event handlers, or browser APIs
Client Components

Client Components

Add "use client" at the top of a file to make it a Client Component. These are needed for interactivity: hooks, event handlers, and browser APIs. 

"use client";

import { useState } from "react";

export function LikeButton({ initialLikes }: { initialLikes: number }) {
  const [likes, setLikes] = useState(initialLikes);

  return (
    <button onClick={() => setLikes(l => l + 1)}>
      ❤ {likes}
    </button>
  );
}

Best practice: keep Client Components small and push them to the leaves of your component tree. Fetch data in Server Components, pass it down: 

// app/posts/page.tsx — Server Component (fetches data)
export default async function PostsPage() {
  const posts = await getPosts();
  return <PostList posts={posts} />; // pass to client
}

// components/PostList.tsx — Client Component (interactive)
"use client";
export function PostList({ posts }: { posts: Post[] }) {
  const [search, setSearch] = useState("");
  const filtered = posts.filter(p => p.title.includes(search));
  return ( ... );
}

Key Takeaways

  • "use client" marks the boundary — everything it imports also becomes client
  • Props passed from Server → Client must be serializable (no functions/classes)
  • Push client boundaries down: keep most code on the server
  • Pattern: Server fetches data → Client handles interaction
Composition patterns
Server + Client boundaries, children pattern, interleaving
▼

Server & Client Composition Patterns

The key to a well-architected Next.js app is knowing where to draw the client boundary. Server Components can render Client Components, but Client Components cannot import Server Components directly. However, you can pass Server Components as children to Client Components. 

// ✗ WRONG — importing a Server Component inside a Client Component
"use client";
import ServerComponent from "./ServerComponent"; // becomes client!

export function ClientWrapper() {
  return <ServerComponent />; // ✗ this is now a client component
}

// ✓ CORRECT — pass Server Component as children (the "donut" pattern)
"use client";
export function ClientWrapper({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false);
  return (
    <div>
      <button onClick={() => setOpen(!open)}>Toggle</button>
      {open && children}  {/* Server Component stays on server! */}
    </div>
  );
}

// app/page.tsx — Server Component composes both
export default function Page() {
  return (
    <ClientWrapper>
      <ServerComponent />  {/* rendered on the server */}
    </ClientWrapper>
  );
}

Common composition patterns:

// Pattern 1: Provider wrapper (context at the root)
// app/layout.tsx
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <ThemeProvider>     {/* Client Component */}
          <AuthProvider>     {/* Client Component */}
            {children}          {/* Server Components! */}
          </AuthProvider>
        </ThemeProvider>
      </body>
    </html>
  );
}

// Pattern 2: Interactive island in a static page
export default async function ProductPage({ params }) {
  const { id } = await params;
  const product = await getProduct(id); // server fetch
  return (
    <article>
      <h1>{product.name}</h1>           {/* static */}
      <p>{product.description}</p>       {/* static */}
      <AddToCartButton                 {/* interactive island */}
        id={product.id}
        price={product.price}
      />
      <ReviewList reviews={product.reviews} /> {/* static */}
    </article>
  );
}

// Pattern 3: Fetch in server, interact in client
export default async function DashboardPage() {
  const data = await getChartData();      // server fetch
  return <InteractiveChart data={data} />; // client renders
}

Key Takeaways

  • Client Components can't import Server Components — but can receive them as children
  • The "donut pattern": Client wrapper with Server content passed through children
  • Keep Client Components as small "islands" of interactivity inside server-rendered pages
  • Props flowing from Server → Client must be serializable (JSON-safe: no functions, Dates, Maps)
  • Context providers are Client Components — wrap them at the layout level, pass pages as children

server-only & client-only Packages

These packages provide build-time guards that throw an error if server code is accidentally imported into a Client Component, or vice versa. This prevents leaking secrets or shipping server-heavy code to the browser. 

# Install the guard packages
npm install server-only client-only
// lib/db.ts — mark as server-only
import "server-only";

export async function getUsers() {
  // This uses DATABASE_URL — a secret
  return db.user.findMany();
}

// If a Client Component tries to import this file:
// ✗ BUILD ERROR: "server-only" module was imported from a Client Component
// This catches the mistake at build time, not in production!
// hooks/useLocalStorage.ts — mark as client-only
import "client-only";

export function useLocalStorage(key: string) {
  // Uses window.localStorage — browser API
  const [value, setValue] = useState(() =>
    localStorage.getItem(key)
  );
  // ...
}

The "taint" API for extra protection:

// Prevent specific objects from being passed to Client Components
import { experimental_taintObjectReference } from "react";

export async function getUser(id: string) {
  const user = await db.user.findUnique({ where: { id } });
  experimental_taintObjectReference(
    "Do not pass user object to client",
    user
  );
  return user;
}
// Now if you accidentally pass `user` as a prop to a
// Client Component, React throws an error at runtime.
// Instead, pick only the safe fields to pass down.

Key Takeaways

  • import "server-only" at the top of files that should never run in the browser
  • import "client-only" for files that need browser APIs (localStorage, window)
  • These are build-time checks — they catch mistakes before they reach production
  • Use for: database queries, API key usage, auth utilities, and any file with secrets
Data fetching patterns
fetch, async components, parallel fetching
▼

Data Fetching Patterns

In the App Router, you fetch data directly in Server Components using async/await. No getServerSideProps or getStaticProps needed. 

// Simple fetch in a Server Component
export default async function ProductsPage() {
  const res = await fetch("https://api.example.com/products");
  const products: Product[] = await res.json();
  return products.map(p => <div key={p.id}>{p.name}</div>);
}

// Parallel fetching — don't await sequentially!
export default async function DashboardPage() {
  // Start ALL fetches at the same time
  const [user, posts, analytics] = await Promise.all([
    getUser(),
    getPosts(),
    getAnalytics(),
  ]);

  return ( ... );
}
Next.js 14 → 15 change: fetch caching changed. In Next.js 14, fetch() was cached by default (force-cache). In Next.js 15, fetch() is not cached by default (no-store). You must opt-in to caching explicitly. 
// Next.js 15+ — explicit caching
const res = await fetch(url);
// ↑ NOT cached (default in v15)

const res = await fetch(url, { cache: "force-cache" });
// ↑ Cached indefinitely (static)

const res = await fetch(url, { next: { revalidate: 3600 } });
// ↑ Cached for 1 hour (ISR)

Key Takeaways

  • Fetch data directly in async Server Components — no special APIs needed
  • Use Promise.all() for parallel fetches — avoid sequential waterfalls
  • Next.js 15 changed fetch defaults: no caching unless you opt-in
  • { next: { revalidate: N } } = ISR (refresh every N seconds)

Caching in Next.js

Next.js has four layers of caching. Understanding them is critical for production apps. 

// 1. REQUEST MEMOIZATION (per-request, automatic)
// Same fetch URL called multiple times in one render
// → only ONE actual network request, result is shared
const user1 = await fetch("/api/user"); // fetches
const user2 = await fetch("/api/user"); // reuses result

// 2. DATA CACHE (persistent, across requests)
// fetch results stored on the server between requests
await fetch(url, { cache: "force-cache" }); // opt-in in v15
await fetch(url, { next: { revalidate: 60 } }); // timed

// 3. FULL ROUTE CACHE (production only)
// Entire rendered HTML + RSC payload cached at build
// Static routes are pre-rendered and served from cache

// 4. ROUTER CACHE (client-side, in-memory)

Streaming & Suspense

Streaming lets you progressively send UI from the server. Fast parts render immediately; slow parts stream in when ready, without blocking the whole page. 

import { Suspense } from "react";

export default function DashboardPage() {
  return (
    <div>
      <h1>Dashboard</h1>  {/* renders instantly */}

      <Suspense fallback={<p>Loading stats...</p>}>
        <Stats />  {/* streams in when data is ready */}
      </Suspense>

      <Suspense fallback={<p>Loading feed...</p>}>
        <Feed />   {/* streams independently */}
      </Suspense>
    </div>
  );
}

// Each async component streams independently
async function Stats() {
  const data = await getStats(); // slow API call
  return <div>{data.visitors} visitors</div>;
}

Key Takeaways

  • Wrap slow async components in <Suspense> with a fallback
  • loading.tsx is just a shortcut for a Suspense boundary around the page
  • Each Suspense boundary streams independently — fast parts aren't blocked
  • This is HTTP streaming — the browser starts rendering before all data is ready

Static vs Dynamic Rendering

Next.js automatically decides whether a route is static (rendered at build time) or dynamic (rendered at request time), based on what your code does. 

// STATIC (build time) — when the page has:
// - No dynamic data fetching (or cached fetches)
// - No dynamic APIs (cookies, headers, searchParams)
// Result: pre-rendered HTML, served from CDN

// DYNAMIC (request time) — triggered by:
// - cookies(), headers() — reading request data
// - searchParams — URL query parameters
// - fetch with no-store — uncached data
// Result: rendered fresh on every request

// Force dynamic rendering:
export const dynamic = "force-dynamic";

// Force static rendering:
export const dynamic = "force-static";

// ISR (revalidate every N seconds):
export const revalidate = 3600; // 1 hour
Next.js 15 — Partial Prerendering (PPR): An experimental feature that combines static and dynamic in ONE page. The static shell is served instantly from CDN, and dynamic parts stream in via Suspense. This is the future of Next.js rendering. 
// next.config.ts — enable PPR
const nextConfig = {
  experimental: { ppr: "incremental" }
};

// app/page.tsx — opt-in per route
export const experimental_ppr = true;

Key Takeaways

  • Next.js auto-detects static vs dynamic — you rarely need to configure it
  • Using cookies(), headers(), or searchParams makes a route dynamic
  • ISR: export const revalidate = N regenerates static pages periodically
  • PPR (Next.js 15+) combines static shells with dynamic streamed content

The use() Hook (React 19)

React 19 introduced use() — a new hook that can read promises and read context inside Client Components. Unlike other hooks, use() can be called inside conditionals and loops. 

// Reading a Promise — unwrap data from the server
"use client";
import { use } from "react";

export function UserProfile({ userPromise }: {
  userPromise: Promise<User>;
}) {
  // use() suspends until the promise resolves
  const user = use(userPromise);
  return <h1>{user.name}</h1>;
}

// Server Component passes the promise down
// app/profile/page.tsx
import { Suspense } from "react";

export default function ProfilePage() {
  // Start fetching immediately — don't await!
  const userPromise = getUser();

  return (
    <Suspense fallback={<p>Loading profile...</p>}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}

Reading context conditionally (impossible with useContext): 

"use client";
import { use } from "react";
import { ThemeContext } from "./theme";

export function ThemedCard({ useTheme }: { useTheme: boolean }) {
  // ✓ use() works inside conditionals — useContext does NOT
  if (useTheme) {
    const theme = use(ThemeContext);
    return <div style={{ background: theme.bg }}>Themed!</div>;
  }
  return <div>Default</div>;
}

// Key differences:
// useContext(ctx)  → must be at top level, can't be conditional
// use(ctx)         → can be inside if/else, loops, try/catch
// use(promise)     → suspends the component until resolved
React 19 / Next.js 15+: use() is a stable API in React 19. It replaces patterns like throw promise (the old Suspense mechanism) with a cleaner, composable API. The promise must be created outside the Client Component (typically in a Server Component) and passed as a prop.

Key Takeaways

  • use(promise) unwraps a Promise inside a Client Component — triggers Suspense
  • use(context) reads context and can be called conditionally (unlike useContext)
  • Pass promises from Server Components → Client Components to start fetching early
  • Always wrap components using use(promise) in a <Suspense> boundary

Optimistic Updates

Show changes immediately in the UI before the server confirms them. If the action fails, the optimistic state automatically reverts. 

"use client";
import { useOptimistic } from "react";
import { toggleTodo } from "./actions";

export function TodoList({ todos }: { todos: Todo[] }) {
  const [optimistic, addOptimistic] = useOptimistic(
    todos,
    (state, id: number) =>
      state.map(t => t.id === id ? { ...t, done: !t.done } : t)
  );

  async function handleToggle(id: number) {
    addOptimistic(id);           // instant UI update
    await toggleTodo(id);        // server action (may take time)
  }

  return (
    <ul>
      {optimistic.map(todo => (
        <li key={todo.id} onClick={() => handleToggle(todo.id)}>
          {todo.done ? "✓" : "○"} {todo.title}
        </li>
      ))}
    </ul>
  );
}

Key Takeaways

  • useOptimistic(data, updateFn) returns optimistic state + setter
  • Call the setter before the server action for instant feedback
  • State auto-reverts if the server action throws an error
  • React 19 feature — works natively with Next.js 15+ Server Actions

useTransition with Server Actions

When calling Server Actions outside of <form> elements (e.g., on button clicks, toggles, or drag events), use useTransition to get a pending state and keep the UI responsive during the mutation. 

"use client";
import { useTransition } from "react";
import { deletePost, publishPost } from "./actions";

export function PostActions({ postId }: { postId: number }) {
  const [isPending, startTransition] = useTransition();

  function handleDelete() {
    startTransition(async () => {
      await deletePost(postId);
      // UI updates after revalidation — no manual setState needed
    });
  }

  function handlePublish() {
    startTransition(async () => {
      await publishPost(postId);
    });
  }

  return (
    <div>
      <button onClick={handlePublish} disabled={isPending}>
        {isPending ? "Publishing..." : "Publish"}
      </button>
      <button onClick={handleDelete} disabled={isPending}>
        {isPending ? "Deleting..." : "Delete"}
      </button>
    </div>

}

// 1. <form action={...}> — for standard form submissions
//    Use useFormStatus() for pending state inside the form
//    Use useActionState() for form state + validation errors

// 2. useTransition — for non-form mutations
//    Button clicks, toggles, drag-and-drop, swipe actions

// 3. Direct await — fire-and-forget (rare)
//    Only when you don't need pending state
//    Example: logging, analytics, background tasks

Key Takeaways

  • useTransition wraps Server Action calls outside of forms — provides isPending

Server Actions Security

Server Actions are HTTP POST endpoints under the hood. Always validate and authorize — anyone can call them. 

"use server";
import { auth } from "./auth";

export async function deletePost(postId: number) {
  // 1. ALWAYS check authentication
  const session = await auth();
  if (!session) throw new Error("Unauthorized");

  // 2. ALWAYS check authorization
  const post = await db.post.findUnique({ where: { id: postId } });
  if (post?.authorId !== session.user.id) {
    throw new Error("Forbidden");
  }

  // 3. ALWAYS validate input
  if (typeof postId !== "number" || postId < 1) {
    throw new Error("Invalid ID");
  }

  await db.post.delete({ where: { id: postId } });
  revalidatePath("/blog");
}

Key Takeaways

  • Server Actions are public endpoints — always authenticate and authorize
  • Next.js provides built-in CSRF protection via encrypted action IDs
  • Validate all inputs server-side (Zod) — never trust client data
  • Use closures carefully — captured variables are sent to the client

Middleware

// middleware.ts (project root or src/)

export function middleware(request: NextRequest) {

    return NextResponse.redirect(
      new URL("/login", request.url)

  return NextResponse.next();

Next.js provides cookies() and headers() functions to read and write HTTP cookies and headers in Server Components, Server Actions, and Route Handlers. Using these makes a route dynamic (rendered at request time).

    httpOnly: true,
    secure: true,
    sameSite: "lax",

export async function GET() {
  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      for (let i = 0; i < 5; i++) {
        controller.enqueue(encoder.encode(`data: chunk ${i}\n\n`));
        await new Promise(r => setTimeout(r, 1000));
      }
      controller.close();
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache",
      "Connection": "keep-alive",
    },
  });
}
// app/api/webhook/route.ts — Webhook handler
import { NextRequest, NextResponse } from "next/server";
import crypto from "crypto";

export async function POST(request: NextRequest) {
  const body = await request.text();
  const signature = request.headers.get("x-webhook-signature");

  // Verify webhook signature
  const expected = crypto
    .createHmac("sha256", process.env.WEBHOOK_SECRET!)
    .update(body)
    .digest("hex");

  if (signature !== expected) {
    return NextResponse.json({ error: "Invalid" }, { status: 401 });
  }

  const event = JSON.parse(body);
  await processWebhook(event);
  return NextResponse.json({ received: true });
}

// CORS — enabling cross-origin requests
export async function OPTIONS() {
  return new Response(null, {
    headers: {
      "Access-Control-Allow-Origin": "*",
      "Access-Control-Allow-Methods": "GET, POST, OPTIONS",
      "Access-Control-Allow-Headers": "Content-Type, Authorization",
    },
  });
}

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;
  const hasLocale = locales.some(l => pathname.startsWith(`/${l}`));
  if (hasLocale) return;

Pages Router vs App Router

The biggest change in Next.js history. The Pages Router (pre-13) and App Router (13+) are fundamentally different architectures. 

// ═══ PAGES ROUTER (legacy) ═══
// pages/index.tsx → /
// pages/about.tsx → /about
// pages/api/users.ts → /api/users

// Data fetching: special exported functions
export async function getServerSideProps() {    // SSR
  return { props: { data: await fetchData() } };
}
export async function getStaticProps() {          // SSG
  return { props: { data: await fetchData() }, revalidate: 60 };
}

// ═══ APP ROUTER (modern) ═══
// app/page.tsx → /
// app/about/page.tsx → /about
// app/api/users/route.ts → /api/users

// Data fetching: just async/await in the component
export default async function Page() {
  const data = await fetchData(); // direct!
  return <div>{data}</div>;
}

Key Takeaways

  • Pages Router: pages/ directory, getServerSideProps/getStaticProps
  • App Router: app/ directory, Server Components, async components
  • App Router adds: layouts, loading/error UI, streaming, Server Actions
  • Both can coexist — migrate incrementally
